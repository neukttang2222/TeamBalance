# Product Scoring Policy v1
문서 상태: 초안 고정안
적용 대상: 제품화 1차 설계
목적: 어뷰징을 줄이면서도 사용자 체감 공정성을 확보하는 기여도 점수 정책 정의

---

## 1. 문서 목적

본 문서는 제품화 이후 TeamBalance의 기여도 점수 산정 기준을 정의한다.

현재 MVP는 다음 핵심 루프를 기준으로 동작한다.

Task 생성
-> 결과물 제출
-> AI 역질문 1개 생성
-> 사용자 답변 제출
-> AI 채점
-> 기여도 점수 반영
-> 대시보드 표시

제품화 이후에도 AI 질문 1개, 답변 1회, AI raw_score 1~5 구조는 유지한다.
다만 점수의 최종 반영 단위를 `submit`이 아니라 `승인된 task`로 변경하여,
중복 제출, 반복 수정, 저가치 task 분할을 통한 점수 부풀리기를 방지한다.

---

## 2. 핵심 원칙

### 2.1 점수는 제출 횟수가 아니라 승인된 기여에 대해 부여한다
- 사용자가 여러 번 제출했다고 해서 점수가 여러 번 누적되지 않는다.
- 본점수(main score)는 각 task당 1회만 잠금(lock) 처리된다.

### 2.2 수정/보완은 새 task가 아니라 같은 task의 버전(version)으로 처리한다
- 같은 업무 범위의 보완 제출은 동일 task의 새 버전으로 관리한다.
- 같은 일을 여러 개 task로 쪼개는 행위는 허용하지 않는다.

### 2.3 이미 잠금된 본점수는 덮어쓰지 않는다
- 최종 승인된 task의 본점수는 이후 수정으로 직접 하향/상향 갱신하지 않는다.
- 승인 이후 추가 작업은 별도 `추가기여점수(delta bonus)` 또는 `child task`로 처리한다.

### 2.4 AI 점수는 중요하지만 전체 점수의 전부는 아니다
- AI raw_score는 사용자의 실제 이해도와 설명력을 검증하는 핵심 지표다.
- 다만 최종 기여도는 AI 점수 하나만으로 확정하지 않고,
  task 기본점수, 승인 여부, 추가기여 여부를 함께 반영한다.

### 2.5 관리자가 요청한 실질 보완은 무보상으로 두지 않는다
- 같은 task의 보완이라고 해서 항상 0점으로 처리하지 않는다.
- manager가 인정한 실질 보완에 한해 제한된 추가기여점수를 부여한다.

---

## 3. 용어 정의

### 3.1 Task
프로젝트 내에서 manager 또는 승인 절차를 통해 점수 대상이 된 업무 단위.

### 3.2 Submission Version
하나의 task에 대해 제출된 각 버전.
예: v1, v2, v3

### 3.3 Main Score
해당 task에 대해 1회만 부여되는 본점수.
대시보드 총점의 핵심 집계값이다.

### 3.4 Delta Bonus
동일 task 내 추가 보완, 추가 조사, 피드백 반영 등 실질 추가노동에 대해
제한적으로 부여하는 추가기여점수.

### 3.5 Child Task
기존 task의 보완 수준을 넘어 실제 업무 범위가 확장된 경우,
manager 승인 하에 새로 생성하는 하위 task.

### 3.6 Locked Score
승인 완료 후 확정되어 더 이상 덮어쓰지 않는 점수.

---

## 4. 점수 단위

본 서비스의 기여도 점수는 아래 세 층으로 나눈다.

### 4.1 본점수(Main Score)
- 각 task당 최대 1회만 부여
- task의 최종 승인 시 잠금 처리
- 대시보드 메인 집계 대상

### 4.2 추가기여점수(Delta Bonus)
- 같은 task 내 실질 보완 작업에 대해 제한적으로 부여
- manager 승인 필요
- 누적 상한 존재
- 대시보드에서 본점수와 분리 표시

### 4.3 활동기록(Activity Log)
- 제출 횟수
- 피드백 반영
- 수정 이력
- retry 이력
- 댓글/검토 이력

활동기록은 공정성 설명에는 활용하되,
핵심 랭킹 점수에 직접 큰 비중으로 반영하지 않는다.

---

## 5. 최종 점수 공식

### 5.1 본점수 공식
`main_score = base_points × approval_factor × ai_factor`

#### base_points
- task 생성 시 system policy 또는 manager가 정한 기본점수
- 사용자가 직접 입력하지 않는다
- task type / task template / manager assignment 정책에 따라 결정한다

권장 기본점수 예시:
- 자료 정리 / QA / 단순 정리: 1
- UI 구현 / 문서화 / 화면 구성: 2
- 핵심 기능 구현 / 설계 / 복잡한 분석: 3

#### approval_factor
- 승인 전: 0
- 승인 완료: 1
- 반려 / 취소: 0

#### ai_factor
AI raw_score를 그대로 배수로 사용하지 않고, 아래와 같이 완만한 계수로 변환한다.

- raw_score 1 -> 0.70
- raw_score 2 -> 0.85
- raw_score 3 -> 1.00
- raw_score 4 -> 1.10
- raw_score 5 -> 1.20

이유:
- AI 답변 1회 결과만으로 전체 기여도가 과도하게 출렁이는 것을 방지한다
- 이해도 차이는 반영하되, 지나친 점수 급등락은 막는다

### 5.2 추가기여점수 공식
`delta_bonus = approved_delta_points`

권장 규칙:
- 한 번의 보완에 대해 0 ~ 0.5점 또는 0 ~ 1점 범위
- 누적 상한: `base_points의 25%` 또는 정책상 고정 상한 중 더 작은 값
- manager 승인 없이 자동 누적되지 않는다

예시:
- base_points = 3인 task
- 추가기여점수 누적 상한 = 0.75
- 보완 제출 3회를 했더라도 총 delta bonus는 0.75를 넘지 못함

### 5.3 최종 반영 점수
`final_contribution_score = locked_main_score + total_delta_bonus`

## 6. 점수 반영 규칙

### 6.1 점수 대상 task 규칙
점수 대상이 되려면 아래 조건을 만족해야 한다.

- 프로젝트에 속한 유효 task일 것
- manager가 생성했거나 승인한 task일 것
- 담당자가 명확할 것
- 중복/유사 task로 판정되지 않을 것
- 취소되지 않았을 것

### 6.2 본점수 잠금 규칙
- task가 최초 승인되면 해당 버전의 점수를 본점수로 잠금 처리한다
- 잠금 이후 동일 task에서 본점수는 직접 변경하지 않는다

### 6.3 수정/보완 규칙
- manager가 “보완 요청”한 경우 같은 task의 새 버전으로 처리한다
- 보완 버전은 기존 잠금 점수를 덮어쓰지 않는다
- 보완 가치가 인정되면 delta bonus만 추가한다

### 6.4 범위 확장 규칙
아래 경우에는 보완이 아니라 새 업무 범위로 본다.

- 최초 task 목표를 명확히 넘어서는 조사/구현/분석
- 추가 산출물이 별도 검토 대상인 경우
- manager가 독립 기여로 판단한 경우

이 경우:
- 기존 task는 유지
- 별도 child task를 manager 승인 하에 생성
- child task는 별도의 main score 대상이 될 수 있다

### 6.5 중복 task 방지 규칙
아래 조건 중 하나에 해당하면 중복 또는 유사 task 후보로 본다.

- 제목이 유사함
- task_goal이 실질적으로 동일함
- 제출 내용 핵심이 반복됨
- 같은 담당자에게 동일 유형 task가 짧은 시간 간격으로 반복 생성됨

중복으로 판정된 task는 아래 중 하나로 처리한다.
- 기존 task에 병합
- child task 전환
- 점수 비대상 처리
- 취소

---

## 7. 공정성 보호 규칙

### 7.1 제출 횟수 자체는 점수 근거가 아니다
- 많이 제출했다고 고득점이 되지 않는다
- “여러 번 시도함”은 기록은 되지만 main score를 늘리지 않는다

### 7.2 낮은 품질의 반복 제출은 점수 이득을 만들지 않는다
- 동일 task 반복 제출은 main score를 다시 지급하지 않는다
- manager 승인 없는 추가 제출은 점수화하지 않는다

### 7.3 수정 요청을 받은 작업자를 무조건 손해 보게 하지 않는다
- manager 요청에 따른 실질 보완은 delta bonus로 일부 보상한다
- 단, 본점수 중복 지급은 하지 않는다

### 7.4 이미 확정된 점수를 뒤집지 않는다
- 승인 완료된 본점수는 원칙적으로 덮어쓰지 않는다
- 이후 추가 작업은 addendum / delta bonus / child task로만 처리한다

---

## 8. 대시보드 표기 규칙

대시보드에는 단일 총점만 노출하지 않는다.
아래 항목을 함께 표시한다.

- 승인된 본점수 총합
- 추가기여점수 총합
- 승인된 task 수
- 보완 요청 후 반영 완료 건수
- 반려/취소 건수

권장 표시 예시:
- Main Score Total
- Delta Bonus Total
- Approved Tasks
- Revision Completed
- Rejected / Canceled

이유:
- “작은 task를 많이 만든 사람”보다
  “승인된 의미 있는 기여를 한 사람”이 더 잘 보이게 하기 위함이다.

---

## 9. 역할별 책임

### 9.1 Worker
- 배정된 task 수행
- 제출 및 보완 수행
- AI 질문 답변

### 9.2 Manager
- task 생성 또는 승인
- 중복/병합 판단
- 보완 요청
- 승인/반려
- delta bonus 승인
- child task 생성 승인

### 9.3 System
- AI 질문 생성
- AI 답변 채점
- 버전 이력 저장
- 잠금 점수 계산
- 대시보드 집계

---

## 10. 예외 처리 원칙

### 10.1 승인 전 점수
승인 전 AI 결과는 `임시 검증점수(provisional score)`로만 본다.
승인 전에는 대시보드 총점에 반영하지 않는다.

### 10.2 승인 후 추가 수정
승인 이후에도 수정은 가능하나 아래 중 하나로만 처리한다.
- delta bonus
- child task
- 운영자 승인 하의 재열기(reopen, 매우 예외적)

### 10.3 reopen 정책
reopen은 아래 조건에서만 허용한다.
- 잘못된 task 매핑
- 명백한 범위 오판
- 운영자 판단상 점수 무효가 필요한 경우

reopen 시에도 과거 점수는 삭제하지 않고 이력으로 남긴다.

---

## 11. 제품 1차 구현 우선순위

### 필수
- task별 main score 1회 잠금
- version 이력
- manager 승인/보완 요청
- delta bonus 상한
- 대시보드 분리 표기

### 후순위
- 중복 task 자동 유사도 탐지
- child task 추천
- 협업/기여 다면 지표 고도화

---

## 12. 최종 정책 한 줄 요약

한 task에는 한 번의 본점수만 부여하고,
수정은 버전으로 남기며,
manager가 인정한 실질 보완만 제한된 추가기여점수로 반영한다.
