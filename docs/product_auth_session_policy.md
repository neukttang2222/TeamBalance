# Product Auth Session Policy

문서 상태: implementation-ready draft
기준 문서:
- `docs/product_api_contract_draft.md`
- `docs/product_authorization_matrix.md`
- `docs/product_team_project_domain.md`

## 1. 문서 목적

이 문서는 제품 레이어의 인증, 로그인 상태, 세션 처리 기준을 고정하기 위한 정책 문서다.

핵심 원칙:
- 인증은 제품 레이어의 기준점이다.
- API body에서 사용자 식별값을 제거한다.
- 서버는 현재 로그인한 사용자를 기준으로 요청을 처리한다.
- 현재 MVP 핵심 루프는 유지하되, 사용자 식별 방식만 제품 기준으로 정리한다.

---

## 2. 인증 목표

제품 1차에서 필요한 인증 목표는 아래와 같다.

- 회원가입
- 로그인
- 로그아웃
- 세션 유지

이 범위면 다음 단계 DB/API/UI 구현에 필요한 최소 인증 기준을 제공할 수 있다.

---

## 3. 현재 MVP 대비 변경점

현재 제품화 기준에서 가장 중요한 변경점은 아래 2개다.

1. `user_id`를 프론트 입력으로 받지 않는다.
2. 서버가 현재 로그인 사용자를 기준으로 요청을 처리한다.

즉:
- task 생성 요청 body에 creator user id를 직접 넣지 않는다.
- submission 생성 요청 body에 submitted_by를 직접 넣지 않는다.
- answer/retry/조회도 현재 세션 사용자 기준으로 권한을 판단한다.

---

## 4. 권장 인증 구조

제품 1차에서는 아래 구조를 권장한다.

- `Supabase Auth` 또는 동등한 managed auth 사용

이유:
- 회원가입/로그인/세션 관리 구현 부담 감소
- 이메일 기반 인증 기본 제공
- 프론트와 서버에서 현재 사용자 식별을 일관되게 처리 가능

대체 가능성:
- 반드시 Supabase만 고정하는 것은 아니다.
- 다만 제품 1차에서는 managed auth 계열을 권장 기준으로 둔다.

---

## 5. Auth Identity와 앱 User Profile 연결

권장 구조:

1. 인증 시스템은 auth identity를 관리한다.
2. 앱 DB는 제품용 user profile을 관리한다.
3. 두 레이어는 공통 user id 또는 auth provider user id로 연결한다.

권장 해석:
- auth identity는 로그인 가능 여부와 세션의 기준
- app user profile은 표시 이름, 제품 내부 활동, 멤버십 연결의 기준

즉, 인증과 제품 프로필을 완전히 섞지 않고 연결 구조로 가져간다.

---

## 6. 최소 사용자 프로필 필드

제품 1차 최소 user profile 필드는 아래를 권장한다.

- `id`
- `email`
- `display_name`
- `created_at`
- `last_login_at` optional

추가 메모:
- `id`는 앱 내부 PK
- `email`은 로그인 계정 식별의 기본값
- `display_name`은 UI 표시용
- `last_login_at`은 선택적 운영 편의 필드

---

## 7. API 처리 원칙

protected API 공통 원칙:
- 인증된 사용자만 접근한다.
- 서버는 현재 인증 세션의 사용자를 기준으로 요청을 처리한다.

멤버십 검사 원칙:
- task 생성/제출/답변/조회는 현재 사용자와 프로젝트 멤버십 기준으로 검사한다.
- 팀 구조는 상위 도메인 기준이지만, 실제 권한 체크는 프로젝트 멤버십을 우선 기준으로 한다.

실무적으로는 아래 순서를 권장한다.

1. 현재 세션 사용자 확인
2. 프로젝트 멤버 여부 확인
3. 프로젝트 역할 확인
4. task 또는 submission 상태 조건 확인

---

## 8. 세션 / 토큰 처리 개념

제품 1차 개념은 아래 수준으로 고정한다.

### 8.1 로그인 후 세션 유지

- 로그인 성공 후 클라이언트는 인증 세션을 유지한다.
- 서버는 요청마다 세션 또는 토큰을 통해 현재 사용자를 복원한다.

### 8.2 토큰 처리

- 액세스 토큰 또는 세션 쿠키 기반 인증 모두 가능하다.
- 제품 1차에서는 사용 중인 managed auth의 기본 권장 방식을 따르는 것을 우선한다.

### 8.3 로그아웃

- 로그아웃 시 클라이언트 세션을 제거한다.
- 서버는 이후 요청을 비인증 사용자로 처리한다.

중요 원칙:
- 세션은 “누가 현재 사용자냐”를 확정하는 기준이다.
- API body로 사용자 identity를 대체하지 않는다.

---

## 9. UI 영향

인증 정책은 UI에 직접 영향을 준다.

### 9.1 로그인 전/후 라우팅

- 로그인 전에는 인증이 필요한 화면 접근을 제한한다.
- 로그인 후에는 사용 가능한 프로젝트 진입 화면으로 이동한다.

### 9.2 현재 사용자 컨텍스트

- 프론트는 현재 사용자 컨텍스트를 전역 상태 또는 세션 컨텍스트로 보유한다.
- 현재 사용자 정보는 API 요청 body가 아니라 세션 컨텍스트에서 파생된다.

### 9.3 프로젝트 선택 진입 흐름

권장 기본 흐름:
1. 로그인
2. 현재 사용자의 팀/프로젝트 목록 로드
3. 마지막 프로젝트 또는 선택 가능한 프로젝트로 진입
4. 첫 기본 화면은 `내 작업`으로 시작

---

## 10. 제품 1차 범위

제품 1차에서 고정할 범위:
- 이메일 기반 회원가입/로그인
- 로그아웃
- 세션 유지
- 현재 사용자 기반 protected API 처리
- 프로젝트 멤버십 기반 권한 검사

---

## 11. 후순위 범위

후순위 항목:
- 소셜 로그인
- 2FA
- 조직 SSO

이 항목들은 제품 1차 필수 범위가 아니다.

---

## 12. 구현 메모

- API 요청 스키마에서 `user_id`, `creator_user_id`, `submitted_by` 같은 클라이언트 직접 입력값은 제거하거나 무시한다.
- 서버는 세션 사용자로부터 creator/submitted_by/actor를 채운다.
- 감사 로그의 actor도 세션 사용자 기준으로 기록한다.

---

## 13. 연결 문구

이 문서는 아래 문서들과 직접 연결된다.

- `product_db_schema_draft`: `creator_user_id`, `submitted_by`, `approved_by`, `actor_user_id`가 세션 사용자 기준으로 채워진다는 원칙을 제공한다.
- `product_api_contract_draft`: protected API와 request body에서 사용자 식별값을 제거하는 기준을 제공한다.
- `product_authorization_matrix`: 역할 검사가 인증 이후 현재 사용자 컨텍스트 위에서 수행된다는 전제를 제공한다.
